## Why

The repo is PLAINTEXT-only. Every big-company HA Kafka on Kubernetes
runs **four** TLS surfaces:

| Surface                       | Protocol | Cert source                          |
|-------------------------------|----------|--------------------------------------|
| Client → Broker (ingress)     | TLS      | Public/internal CA, SAN=`<svc>.kafka` |
| Broker ↔ Broker (replication) | mTLS     | Internal CA, SAN=`<pod>.headless`    |
| Broker → Controller (KRaft)   | mTLS     | Internal CA, SAN=`<pod>.headless`    |
| Controller ↔ Controller       | mTLS     | Internal CA, SAN=`<pod>.headless`    |

The Bitnami chart supports this through `tls.type: PEM` +
`tls.autoGenerated.engine: cert-manager`, but **nobody has shipped a
turnkey values file + cert manifests + rotation runbook** for it.

This contribution delivers exactly that, modelled after what
Confluent Cloud, Stripe, and Shopify run in production.

## Chart keys to override (full diff vs. `values.yaml`)

| Key                                | Value                              | Why                                                 |
|------------------------------------|------------------------------------|-----------------------------------------------------|
| `listeners.client.protocol`         | `TLS`                              | client→broker ingress                               |
| `listeners.controller.protocol`     | `SSL`                              | broker↔controller + controller quorum               |
| `listeners.interbroker.protocol`    | `SSL`                              | broker↔broker replication                           |
| `tls.type`                         | `PEM`                              | Bitnami's preferred type for cert-manager           |
| `tls.autoGenerated.engine`          | `cert-manager`                     | dynamic cert provisioning                           |
| `tls.autoGenerated.enabled`        | `true`                             | turn on auto-cert path                              |
| `tls.serverCerts`                  | see below                          | one entry per listener pointing at a cert-manager Certificate |
| `tls.clientCerts`                  | see below                          | optional mTLS client certs                          |
| `tls.pemChainIncluded`             | `true`                             | chain must include the CA for clients to validate  |
| `auth.client.ssl.existingSecret`   | (omitted)                          | we use only the cert-manager-issued cert            |

Example `tls.serverCerts` (referenced from `certs/02-broker-server.yaml`):

```yaml
tls:
  type: PEM
  autoGenerated:
    enabled: true
    engine: cert-manager
  pemChainIncluded: true
  serverCerts:
    - name: "kafka-server"
      secretName: "kafka-server-cert"     # produced by cert-manager
      dnsNames:
        - "external-kafka"
        - "external-kafka.kafka.svc.cluster.local"
        - "*.external-kafka-controller-headless.kafka.svc.cluster.local"
      ipAddresses: []                    # optional SAN list for NodePort
```

`certs/02-broker-server.yaml` (cert-manager Certificate template):

```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: kafka-server
  namespace: kafka
spec:
  secretName: kafka-server-cert
  issuerRef:
    name: kafka-ca-issuer
    kind: Issuer
  commonName: external-kafka.kafka.svc.cluster.local
  dnsNames:
    - external-kafka
    - external-kafka.kafka.svc.cluster.local
    - "*.external-kafka-controller-headless.kafka.svc.cluster.local"
  duration: 2160h                         # 90 days
  renewBefore: 720h                       # rotate 30 days before expiry
  usages:
    - digital signature
    - key encipherment
    - server auth
    - client auth                          # mTLS: cert is valid for both
```

## Companion files

### `kafbat-values-tls.yaml` (Kafbat with truststore)

```yaml
yamlApplicationConfig:
  kafka:
    clusters:
      - name: external-kafka
        bootstrapServers: external-kafka:9093           # TLS port
        securityProtocol: SSL
        ssl:
          truststoreLocation: /opt/kafka-ui/truststore.jks
          truststorePassword: changeit
        properties:
          ssl.endpoint.identification.algorithm: HTTPS
  auth: { type: disabled }

extraVolumeMounts:
  - name: truststore
    mountPath: /opt/kafka-ui
    readOnly: true
extraVolumes:
  - name: truststore
    secret:
      secretName: kafka-ca-cert              # CA cert exported from cert-manager
```

### `scripts/rotate-certs.sh` (production rotation runbook)

```bash
#!/usr/bin/env bash
set -euo pipefail
# Force-rotate a cert-manager Certificate by deleting its Secret;
# the Issuer will re-issue and the Bitnami chart will hot-reload.
#
# Usage: ./scripts/rotate-certs.sh <cert-name>
CERT="${1:?usage: $0 <cert-name>}"
NS="${NAMESPACE:-kafka}"
kubectl delete secret -n "$NS" "$CERT-cert" --ignore-not-found
echo "Awaiting cert-manager re-issue ..."
kubectl wait --for=condition=Ready -n "$NS" "certificate/$CERT" --timeout=120s
echo "Broker will hot-reload within ~60s. Verify with:"
echo "  ./scripts/verify-tls.sh"
```

### `tests/tls_smoke.go` (Go smoke test using `kafka-go`)

Asserts:
1. Connect with TLS to `external-kafka:9093` and list topics.
2. Connect with mTLS (client cert) and produce a message.
3. Connect without TLS — connection is **rejected**.

## README changes

Insert after the **Deployment modes** section:

```markdown
## Securing the cluster (TLS / mTLS)

Production Kafka on Kubernetes requires TLS for every listener.
This repo ships a turnkey cert-manager-driven bundle that provisions
and rotates four sets of certificates:

- Client-facing broker cert (TLS)
- Inter-broker / broker↔controller certs (mTLS)
- Controller quorum certs (mTLS)

See [todo/01-cert-manager-tls.md](./todo/01-cert-manager-tls.md) for
the design and `values-prod-tls.yaml` for the values file.

```bash
# 1. Bootstrap a dev CA (in prod, use your real Vault/Step-CA Issuer)
kubectl apply -k certs/

# 2. Install the TLS-enabled cluster
helm install tls-kafka oci://.../bitnamicharts/kafka \
  --namespace kafka -f values-prod-tls.yaml --wait --timeout 15m

# 3. Verify
./scripts/verify-tls.sh
```
```

## Open questions to resolve before implementation

1. **Do we use cert-manager's default self-signed Issuer in dev, or
   require the user to bring their own Vault/Step-CA Issuer?**
   *Recommendation:* dev gets a self-signed `Issuer`; the README
   documents swapping `issuerRef` to point at the user's CA.
2. **SAN for cross-region / external access?** If the user is using
   `externalAccess.service.type: LoadBalancer`, we need a wildcard
   SAN (`*.kafka.example.com`). Out of scope for v1; documented as a
   follow-up.
3. **Do we ship a one-shot `kafbat-values-tls.yaml` or do we expect
   users to merge the TLS block into their own?** *Recommendation:*
   ship a one-shot, but make it minimal (3 lines of truststore
   config).
